Section 1
Architecture Overview
The platform is designed to run AI agents that can autonomously plan, act, and complete multi-step tasks. It has five major zones, each with a distinct role — from accepting inbound traffic to governing how the platform is built and changed.
Zone 1

Channel & Access — The front door

All external traffic enters here. Authentication via SSO/OIDC, rate limiting, WAF protection. User identity is propagated as immutable context downstream.

API Gateway SSO / OIDC Rate Limiting Messaging Gateway WAF
Zone 2

Orchestration Engine — The brain (four sub-layers)

2A: Agent Gateway (intent classification, routing, session hydration) · 2B: Durable Execution (state machines, approval gates, HITL) · 2C: LLM Layer (model routing, caching, metering) · 2D: Tool Layer (single outbound boundary for all agent actions)

Intent Classifier ReAct Loop State Machines Approval Gates Tool Gateway Model Router
Zone 3

Memory, Knowledge & Data — Institutional intelligence

Agents know more than the current conversation. Vector store for semantic search, Knowledge Graph for document relationships, Skill Library for reusable procedures, Structured Data Store for business rules and thresholds.

Vector Store Knowledge Graph Skill Library Structured Data Store DataOps
Zone 4

Guardrails & Governance — Non-negotiable safety net

Wraps all zones. Input guards, output guards, step guards, HITL engine, agent identity management, and evaluation. Enforced regardless of workflow config — the hard stops that protect against misconfiguration.

Input Guards Output Guards Step Guards HITL Engine Agent Identity
Zone 5

Infrastructure, Ops & Observability — Full traceability

Every agent action is traced, metered, and auditable. Agent tracing, immutable audit log, Cost/FinOps per session, IaC provisioning, pre-built CI/CD pipelines.

Section 3
Use Case — Invoice 3-Way Match & ERP Posting
A concrete walkthrough of the architecture handling a real accounts payable workflow: verifying an invoice against a PO, Delivery Note, and Goods Receipt Note, then posting to Xero and updating GL ledgers.
ZONE 1

Invoice received

INV-2891 from ABC Supplies Pte Ltd for SGD 48,500 arrives via AP inbox. SSO identifies receiving entity. Invoice parsed and tokenised.

ZONE 2A

Agent Gateway

Intent: accounts_payable / 3-way-match / erp_posting. Agent selected: AP-Matching-Agent v2.1. Session created. Tools bound: ERP-Connector, DocStore-Connector, GL-Writer.

STATE 1–4

3-Way Match

PO-7741: SGD 49,200 within 2% tolerance ✓ · DN-3310: 12 SKUs delivered ✓ · GRN-0892: 12 SKUs receipted ✓

GATE

Approval Gate

SGD 48,500 exceeds auto-approve limit (SGD 10,000). Workflow paused. AP Manager notified and approves. Workflow resumes.

STATE 5–6

Post to Xero + GL

Xero Bill BILL-XR-00421 created. GL journal posted: Dr Inventory SGD 48,500 / Cr AP (ABC Supplies) SGD 48,500. Double-entry balanced.

ZONE 5

Full audit trail

Every state transition, tool call, HITL decision, and GL entry immutably recorded. Full ReAct reasoning trace captured — explainable to auditors.

Key design insight

The agent never directly commits financial transactions. It builds the payload, and a deterministic Completion Gate fires it after rules-based verification. The LLM reasons — the gate acts.

Section 4
Where Approval Gate Rules Are Stored
Rules are split across three zones, each handling a different type. They are evaluated in sequence at runtime. Keeping them separate means each team can update their layer independently.

Zone 2B — Structural rules

  • Which steps require a gate
  • Approver role (ap_manager, cfo)
  • Timeout duration and escalation path
  • Owned by: engineering team
  • Changes: infrequently via CI/CD

Zone 3 — Business rules & thresholds

  • Monetary thresholds by entity and level
  • Delegation of Authority (DoA) matrix
  • Risk flags: first-time vendor, variance, duplicate
  • Owned by: finance and ops teams
  • Changes: via Admin UI with maker-checker

Zone 4 — Hard stops (non-negotiable)

  • Agent identity scoped — AP agent cannot reach payroll
  • Absolute ceiling enforced regardless of workflow config
  • GL payload must balance (Dr = Cr) before Xero call fires
  • Owned by: security and compliance team
Zone 3 Update Pathways

1. Admin UI: Finance team updates thresholds via governed portal — maker-checker enforced, second approver required.  |  2. Automated ingestion: Every completed transaction feeds back automatically via DataOps pipelines.  |  3. Workflow capture: Agent observes HITL decisions, proposes rule updates — humans promote, never the agent.  |  4. Engineering deployment: Skill Library and schema changes via full CI/CD with code review.

Section 5
The Tool Gateway
Two gateways, two directions. The API Gateway handles all inbound traffic from external users into the platform. The Tool Gateway handles all outbound traffic from agents to any target system — internal or external.

Zone 1 — API Gateway

INBOUND · external → platform
  • Authenticates all inbound traffic
  • AuthN, rate limiting, DDoS protection
  • Routes to services or agent gateway
  • Owns: user identity, session tokens
  • Direction: INTO the platform

Zone 2D — Tool Gateway

OUTBOUND · agents → any system
  • Governs all agent-to-system calls
  • Dynamic tool selection at runtime
  • Per-agent credential scoping
  • Owns: service credentials, tool manifests
  • Direction: OUT from the platform
Critical security principle

The agent never holds credentials. The Tool Gateway owns all service accounts, API keys, and OAuth tokens — for both internal and external systems. Break-glass revocation at the gateway level instantly cuts off all agents from a target system in one action. The same Xero instance can be reached by multiple agents, each with different scoped credentials — AP agent cannot access payroll even though both are in Xero.

Credentials held by the Tool Gateway

External SaaS
  • Xero ERP — OAuth 2.0, scoped to Bills.Write, Accounts.Read only
  • Salesforce CRM — OAuth 2.0, vendor records read-only
  • Email / O365 — scoped to AP inbox read, send notifications
  • Group-IB threat intel — API key, read-only feed
Internal systems
  • DocStore — service token, PO/DN/GRN read, AP workflow only
  • ERP internal DB — DB service account, read-only vendor master
  • HR system — service token, approver role lookup only
  • Identity provider — service token, group membership lookup
Infrastructure
  • Zone 3 Vector Store — internal mTLS, read queries only
  • Zone 3 Knowledge Graph — internal mTLS, read traversal
  • Zone 5 Audit Log — internal mTLS, append-only write
  • Message queue — internal mTLS, scoped topics only
Section 6
Integration Pattern Evolution
Organisations don't choose a pattern — they grow into each one. The trigger is always pain. Each pattern layers on top of the previous; you rarely rip out what came before.
P1

Direct service-to-service calls

1–10 services · 2–5 integrations

Each service calls external systems directly. Credentials in environment variables per service. Correct and fast at startup scale.

Trigger to move → credential sprawl, no central audit, rate limit conflicts
P2

ESB / Integration middleware

10–100 services · 10–50 integrations

MuleSoft, Boomi, IBM MQ centralises all calls. Credentials centralised, protocol translation handled. Dominant in large enterprises and regulated industries.

Trigger to move → ESB bottleneck, microservices teams need autonomy, licensing cost
P3

API Gateway outbound proxy

100+ services · cloud-native

API Gateway (Kong, Apigee) handles outbound. Service mesh (Istio, Linkerd) for internal east-west traffic. Teams own integrations with central governance.

Trigger to move → agents making autonomous runtime decisions that P3 can't govern
P4

Tool Gateway (agentic)

N agents · dynamic at runtime

Dedicated outbound gateway for agent-driven calls. Dynamic tool selection, per-agent credential scoping, reasoning-step audit. Does not replace P1–P3 — sits on top.

Design this in from day one if agents are on your 24-month roadmap
Greenfield best practice (2025–2026)

Build in this order: 1. Secrets Manager · 2. Identity Platform · 3. API Gateway (inbound P3) · 4. Tool Gateway (P4 — even if empty) · 5. Service Mesh (when 5+ services) · 6. Observability + Audit Log. Building P4 when agents arrive on a live P3 platform is expensive. Building it empty from day one costs almost nothing.

Section 7
Hybrid Architecture — Two Planes, One Foundation
For a platform that requires both application-to-application integration and agentic workflows simultaneously: keep the planes separate, share the infrastructure underneath. The caller identity determines the plane — not the target system.

(a) Service Integration Plane

App-to-app · service mesh + API GW outbound
  • Caller: deterministic service with fixed identity
  • Call sequence: pre-wired at build time
  • Fails fast — retry logic is deterministic
  • Audit: did the call succeed?
  • Governed by: service mesh + API GW proxy
  • Traffic: AP-Service → GL-Service → Xero

(b) Agent Execution Plane

Agentic workflows · Tool Gateway outbound
  • Caller: autonomous agent deciding at runtime
  • Call sequence: decided dynamically by ReAct loop
  • Failures need context — what was the agent doing?
  • Audit: why was this call made?
  • Governed by: Tool Gateway + Zone 4 Guardrails
  • Traffic: AP-Agent → Tool GW → Xero connector → Xero
Shared Foundation — used by both planes
Secrets Manager — one credential store · auto-rotation Identity Platform — human + agent principals · unified IAM Observability + Audit — unified trace across both planes
The routing rule

Xero can be called by both AP-Service (service plane) and AP-Matching-Agent (agent plane). The service call goes through the API GW outbound proxy. The agent call goes through the Tool Gateway. Same destination — completely separate governance. Same credential from the same Secrets Manager — but delivered through the plane appropriate to the caller.

When an agent calls an internal service

Service mesh is sufficient when

  • Read-only, low-risk internal call
  • High frequency, low governance needed
  • Agent identity propagated via mesh headers

Tool Gateway always when

  • Any write action
  • High-value or regulated call
  • Reasoning-level audit required
Section 8
Key Governing Principles
The distilled principles from the full architecture discussion — the ideas that should guide every design and implementation decision.
Tool Gateway

The Tool Gateway is the only point where agent intent becomes a real-world side effect. Everything before it is reasoning. Everything after it is action. That single boundary is what makes the system auditable, securable, and safe to run autonomously.

Zone 3 Updates

Zone 3 is read-only to agents at runtime. The frequency and formality of each update pathway matches the risk level of the data being changed — a threshold update needs a CFO counter-sign, a new invoice in the vector store is automated.

Hybrid Architecture

Separate the planes, share the foundation. The caller identity determines which plane a call travels through — not the target system, not the data type, not the business domain.

Greenfield Design

Pattern 3 is still the right answer for human-driven integration. But designing only for Pattern 3 is now a deliberate choice to incur technical debt — because agents are not a question of if, only when.

Pattern Evolution

You do not choose a pattern — you grow into each one. The trigger is always pain. Each pattern layers on top of the previous — you rarely rip out what came before.

Approval Gates

Zone 2B holds the structure (when to check). Zone 3 holds the values (what the thresholds are). Zone 4 holds the non-negotiables (what can never happen). Keeping these separate means each team can update their layer without touching the others.
